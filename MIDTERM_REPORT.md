### 1.2 MCP Server 设置

项目采用了 **Model Context Protocol (MCP)** 的 **integrated_browser server**，用于在需要时辅助页面效果预览。具体设置如下：

```
MCP Server: integrated_browser
可用工具:
  - browser_navigate / browser_navigate_back
  - browser_tabs (列出/切换标签页)
  - browser_snapshot / browser_take_screenshot
  - browser_click / browser_hover / browser_type
  - browser_select_option / browser_press_key
  - browser_get_attribute / browser_scroll
  - browser_console_messages / browser_network_requests
  - browser_wait_for / browser_evaluate (可运行 JS)
```

**使用方式**：当开发者对 UI 布局/颜色有疑问时，让智能体先在浏览器中启动本地服务，通过 `browser_snapshot` 获取当前页面结构，再根据输出调整 Qt 样式表（QSS）。虽然 Qt Widgets 本身不跑在浏览器里，但这套流程在**调试颜色/布局比例等视觉参数**时仍有参考价值。

### 1.3 规则文档与技能文档的编写

项目在 `PLAN.md` 中维护了**一份全局规则文档**，它既是人类开发者的设计说明，也是 AI 助手的上下文锚点。文档结构如下：

```
PLAN.md
├── 游戏规则
│   ├── 基本概念（双方方格、发射点、函数图像、命中判定）
│   ├── 障碍物系统（随机分布、最小中心距、破坏规则）
│   ├── 胜负条件
│   ├── 开始界面（NEW GAME / Load / Manage Saves）
│   ├── 方格布局（随机生成规则、半区约束）
│   ├── 资源点数系统（availablePoints 公式 + 各语法单元消耗表）
│   ├── 历史轨迹
│   ├── 暂停系统（ESC 触发、三槽位存档）
│   └── 存档系统（JSON 字段清单、跨会话持久化）
├── 技术栈（C++17 / Qt6 / CMake / Ninja / vcpkg / MVVM）
├── 项目结构（MVVM 各层文件清单与职责划分表）
├── ViewModel-View 接口清单
└── 各实现细节（表达式解析器、碰撞检测、动画帧步、序列化）
```

**编写策略**：
- 代码层面的任何新增/修改都要求同步更新 `PLAN.md` 的对应章节，让文档始终反映最新架构。
- 文档使用**表格+层级列表**，便于 AI 助手做结构化检索。
- 文档中不写"伪代码"，而是直接描述行为约束（如"方块之间最小间距 3.0 单位"），让 AI 在实现时根据约束自行推导代码。

### 1.4 每轮对话的记录与工作流

团队采用的 **"需求 → 规划 → 实现 → 验证"四步对话流**：

**典型一轮对话示例**：

| 步骤 | 人类输入 | AI 输出 | 后续动作 |
|------|---------|---------|---------|
| ①需求 | "把业务逻辑、数据计算和存储都移动到 model" | 回答：现有结构评估 + 迁移方案 | 无代码修改 |
| ②规划 | —（AI 已自行拆解）— | TodoList：重写 GameModel.h → 迁移 SaveManager → 瘦身 GameViewModel → 更新 CMakeLists | 确认后进入实现 |
| ③实现 | —（AI 逐步调用 Edit/Write）— | 逐文件修改：GameModel.h/cpp、SaveManager、GameViewModel.h/cpp、CMakeLists.txt | 每一步可在 IDE 中看到文件差异 |
| ④验证 | —（AI 调用 Shell）— | `cmake --build` 构建日志；若有错误，自动按错误信息返工 | 构建通过后本轮结束 |

**关于对话记录**：所有对话均通过 IDE 的 AI 助手界面进行，自然保留在会话历史中。关键架构决策（如 MVVM 分层边界、点数计算公式、存档 JSON 字段）会同步写到 `PLAN.md`，因此即便对话历史被清理，项目核心规则仍有持久化记录。

### 1.5 实际效果与现存问题

**实际效果（已验证可工作）**：

1. **代码生成速度**：一个典型 UI 页（含布局 + 信号槽 + QSS 样式）从 0 到可编译，约 3-5 轮对话；纯逻辑类（如表达式解析器）一轮可完成。
2. **重构能力**：将 MVVM 架构从"业务逻辑在 ViewModel"重构为"业务逻辑在 Model + ViewModel 薄协调层"，涉及 6 个文件、约 500 行代码的同步迁移，AI 在 1 小时内完成且一次性通过构建。
3. **跨文件一致性**：PLAN.md 作为上下文锚点，AI 能在修改 `GameModel.cpp` 时自动同步调整 `GameViewModel.cpp` 的转发逻辑，避免遗漏信号绑定。

**现存问题**：

1. **信号链遗漏**：当 Model → ViewModel → View 有多层 signal 转发时，AI 偶尔会漏掉最后一段绑定（如 `GameViewModel::trajectoryUpdated` 到 `GameCanvas::update()`），导致"弹道算出来但屏幕没刷新"的问题。
2. **资源文件部署理解不足**：Qt Multimedia（QMediaPlayer）播放 m4a 需要额外的 FFmpeg DLL 和 `plugins/multimedia/` 目录结构，AI 对"vcpkg 会自动部署 DLL 但不创建插件子目录"的理解存在偏差，导致音频功能迟迟未能打通。
3. **跨平台路径大小写**：在 Windows 下 `QFile(":/bgm.m4a")` 与 `QUrl("qrc:/bgm.m4a")` 行为不同，AI 容易混用这两种写法，引入"有时响有时不响"的问题。
4. **MCP Browser 的局限性**：Qt Widgets 应用不跑在浏览器中，integrated_browser server 无法直接渲染 Qt 窗口，所以它的截图/点击能力对本项目帮助有限。团队主要依赖 IDE 的 CMake 构建日志和手动运行来验证效果。

## 二、团队协作情况

### 2.1 团队成员分工

| 成员 | 角色 | 主要职责 |
|------|------|---------|
| **成员 A** | 架构师 + 后端 | 定义 MVVM 分层边界；负责 Model 层（GameModel、SaveManager、JSON 序列化）；碰撞检测/弹道动画算法 |
| **成员 B** | 前端 + UI | 负责 View 层（MainWindow、GameCanvas、ConfigPage、PauseMenuPage）；QSS 样式表编写 |
| **成员 C** | 工具层 + 测试 | 负责 utils/parser（Tokenizer、Parser、Expression、Evaluator）；验证点数计费公式；编写构建脚本 |
| **成员 D** | 构建 + 部署 | CMakeLists.txt / CMakePresets.json / vcpkg.json 维护；跨平台构建测试（Windows / Linux / macOS） |

**所有成员均承担与 AI 助手的交互工作**：每人轮流负责一个子模块，在自己负责的模块中主导与 AI 的对话，其他成员负责 Code Review。

### 2.2 人与智能体协同工作的具体实践

团队总结出三种行之有效的协作模式：

#### 模式一："人类定规则，AI 写代码"
- **流程**：成员 A 在 `PLAN.md` 中写好"资源点数系统"章节（含公式、各语法单元消耗表）→ 成员 C 与 AI 对话实现 `Evaluator` 的 `calculateCost()` → AI 输出代码 → 成员 C 做 Code Review → AI 根据 Review 反馈修正。
- **效果**：人类花 15 分钟写规则文档，AI 花 3 分钟生成 200 行 C++ 代码，人类再花 10 分钟 Review/调 bug，**总时间不到 30 分钟**，远快于传统手写。

#### 模式二："先跑通再重构"
- **流程**：早期版本中，业务逻辑全写在 `GameViewModel.cpp`（单文件 600+ 行，难以维护）→ 成员 A 发起重构需求 → AI 用"TodoList 拆解 + 逐文件迁移"的方式推进 → 每迁移一段就运行 `cmake --build` 验证。
- **效果**：重构过程没有引入新的编译错误，**逻辑行为零回归**。原因是 AI 每次只做"机械迁移 + 接口不变"的操作，不做语义变更。

#### 模式三："AI 写初稿，人类做终审"
- **流程**：成员 B 让 AI 先写一个 `ConfigPage` 的初稿（布局 + 控件命名）→ AI 生成的版本可能样式粗糙 → 成员 B 人工调整 QSS 样式（颜色/字体/边距等视觉效果）→ 再让 AI 基于调整后的版本做增量扩展（添加新配置项）。
- **效果**：视觉/审美类工作（人类强 AI 弱）和机械布局类工作（AI 强人类烦）各归其位，**1 天内完成 5 个页面**的从零搭建。

### 3.2 核心架构 — MVVM 严格分层

**架构图（当前版本已完整实现）**：

```
              ┌──────────────────────────────────┐
              │         View（View 层）           │
              │  MainWindow · GameCanvas         │
              │  FunctionInput · ConfigPage      │
              │  SaveManagerPage · PauseMenuPage │
              │  （纯渲染 + 用户输入）            │
              └──────────────┬───────────────────┘
                             │ signals/slots
                             │ （GameViewModel 暴露的只读接口）
              ┌──────────────▼───────────────────┐
              │      ViewModel（ViewModel 层）    │
              │        GameViewModel              │
              │  （薄协调者：转发 Model → signals）│
              └──────────────┬───────────────────┘
                             │  #include "model/*"
              ┌──────────────▼───────────────────┐
              │         Model（Model 层）         │
              │  GameModel · GameConfig          │
              │  Player · Square · GamePhase     │
              │  SaveInfo · SaveManager          │
              │  （业务逻辑 + 数据 + 序列化）      │
              └──────────────┬───────────────────┘
                             │  调用工具层
              ┌──────────────▼───────────────────┐
              │        Utils（工具层）            │
              │  Geometry · parser/*             │
              │  （纯算法，无业务逻辑）            │
              └──────────────────────────────────┘
```

**关键代码量分布**：

- `src/model/`：≈ 700 行（核心业务逻辑）
- `src/viewmodel/`：≈ 250 行（信号转发 + 细粒度访问器）
- `src/view/`：≈ 1,200 行（UI 渲染 + 样式表逻辑）
- `src/utils/parser/`：≈ 550 行（表达式解析/AST/求值/计费）
- `CMakeLists.txt / vcpkg.json / PLAN.md`：≈ 150 行（配置/文档）

## 四、总体心得与个人感悟

### 4.1 总体心得

**1. AI 不是写代码的替代品，而是让你有精力思考更高层的架构**

在本项目中，我们最直观的体会是：AI 让**写代码这件事本身不再是瓶颈**。真正的瓶颈转而去了"你能不能把需求说清楚"。

比如在最开始的版本中，团队让 AI 直接写"一个双方对战的数学函数游戏"，结果产出一个**600 行单文件版**——能跑，但难以维护。后来我们花了整整**一个下午**做的事情不是写代码，而是写 `PLAN.md`：把 MVVM 每一层的职责边界、数据流向、信号名称、序列化字段一条条写清楚。当这份规则文档完善后，AI 再按照文档重新实现时，**一次性通过构建，且后续扩展（添加暂停系统、添加存档系统、添加颜色配置）几乎零返工**。

**结论：与 AI 协作，前期写规则文档的投入回报率极高。**

**2. MVVM + AI 是绝配**

MVVM 的核心思想是**让状态和行为集中在 Model 层，让 View 层保持"无知"**。这个思想恰好和 AI 工作流完美契合：
- 当 AI 修改 GameModel 时，它不需要知道任何 UI 代码，只要信号签名不变，View 自然能工作。
- 当 AI 修改 View 层时，它不会触碰游戏逻辑，减少引入逻辑 bug 的风险。

我们在"将业务逻辑迁移到 Model"那次重构中深刻感受到这一点：修改涉及 6 个文件、约 500 行代码，**但构建一次通过、逻辑零回归**。这在手写代码时代几乎不可能做到这么干净。

**3. 跨平台部署是 AI 的弱项**

AI 可以轻松写出 "setSource(QUrl(...))" 这样的播放代码，但它**很难自动意识到** Qt Multimedia 在 Windows 上还需要 FFmpeg 的 5-6 个 DLL 和 `plugins/multimedia/` 目录结构。这类"需要本地环境知识 + 底层平台细节"的问题，AI 往往需要多轮试错才能定位。

团队的解决方式是：**把这类平台特定知识写入 `PLAN.md`**，让后续对话中 AI 能直接参考。

**4. 信号槽机制需要人类时刻保持警觉**

Qt 的信号槽机制非常优雅，但也非常容易出现"信号发射了但没有被订阅"的问题。AI 在处理多文件联动（Model 发射信号 → ViewModel 转发 → View 订阅 → Canvas 重绘）时偶尔会漏掉最后一环。这是我们遇到的**最常见、最隐蔽**的 bug 类型。

我们的对策：每次新增一个信号，**强制要求同时列出它的完整订阅链**——从哪个类发出，经过哪些类转发，最终由哪个 View 组件响应。这份清单同样写到 `PLAN.md` 中。

### 4.2 个人反思与成长收获

**成员 A（架构师 + 后端）——**

> "这次项目让我重新理解了'架构'这个词。以前我觉得架构就是文件怎么组织、类怎么继承。但现在我意识到，**架构的核心是给 AI（以及队友）画一个清晰的边界图**：哪些逻辑必须在 Model，哪些只能在 View，哪些信号必须显式转发。当边界不清时，AI 会毫不犹豫地写出违反分层的代码，短期跑得通但长期会拖垮项目。
>
> 最有成就感的时刻是 MVVM 重构完成的那一次 build：看着 20 多个编译单元一个接一个通过，没有一条警告，那种感觉比任何单个算法 debug 都更让人兴奋。"

**成员 B（前端 + UI）——**

> "我负责 View 层。和 AI 配合做 UI 的体验非常有趣：AI 很擅长生成 '骨架代码'（布局嵌套、QSS、信号槽绑定），但在审美上非常不可靠——它经常把按钮间距设置得过大或过小、颜色搭配刺眼。
>
> 我的工作流程变成：**先让 AI 写初稿 → 我手工调一遍颜色/字号/间距 → 让 AI 照着调过的版本做扩展**。这个 'AI 搭骨架、人类穿衣服' 的模式让我们的页面质量远超预期。
>
> 最大的收获是：**不要让 AI 做审美决策**，但也不要自己花时间写重复的布局代码。

**成员 C（工具层 + 测试）——**

> "我负责表达式解析器。写 Tokenizer/Parser/AST/Evaluator 这种东西非常适合 AI——规则明确、边界条件清楚。但我发现一个有趣的现象：**AI 写的 '正确代码' 往往比人类写的更冗长**，它倾向于显式处理每一个分支，而人类可能会用一个查表或模板来合并。
>
> 所以我的工作变成：**先让 AI 生成一个 '一定正确但略笨' 的版本 → 然后我做一轮 refactor，把重复逻辑合并为查表 → 再让 AI 对我重构后的版本做扩展**（比如添加新的三角函数）。
>
> 这个思路让我们的解析器既有正确性保障，又有可维护性。另外，点数计费系统是我们最早发现 AI 容易算错的一个模块——我人肉验证了 3 次，每一次都发现 AI 漏了某个运算符的点数。这让我相信：**涉及数值计算/公式推导的地方，人类永远需要自己算一遍**。"

**成员 D（构建 + 部署）——**

> "我负责 CMake 和跨平台。我最大的体会是：**vcpkg + CMakePresets.json 是跨平台构建的圣杯，但你必须自己吃透它的规则**。AI 可以帮你写出基本的 CMakeLists，但它不知道你的项目在 Windows 上需要 `WIN32`，在 macOS 上需要 `MACOSX_BUNDLE`，在 Linux 上需要 `RPATH`——这些平台条件分支需要人类来指定。
>
> 最让人崩溃的是 Qt 插件部署这件事。vcpkg 的 z-applocal 特性会自动把 Qt DLL 复制到 build 目录，但它**不会创建 `plugins/multimedia/` 子目录结构**。这个细微的差异让我们的音频功能卡了整整一个晚上——没有报错，只是'没声音'。
>
> 这次经历让我学到：**对于没有错误信息的静默失败，AI 的调试能力远弱于人类直觉**。它需要你先把问题定位到'缺少某个 DLL'，才能帮你写出正确的部署脚本。

## 结语

截至 **Fri Jul 10 19:14:17 2026 +0800**，Graphwar 项目已完成核心玩法的全链路实现和可运行的构建版本。它不是一个"玩具项目"——从 MVVM 严格分层、表达式解析器、碰撞检测、动画系统、JSON 序列化存档、跨平台部署来看，它的工程复杂度已经接近小型商业桌面应用。

**AI 给这个项目最大的礼物不是'更快地写代码'，而是'让团队有机会在有限时间内完成更具工程质量的架构'**。在过去，一个 2 周的 C++ 短学期项目通常只能产出"跑起来但架构一团糟"的代码。但在 AI 的辅助下，我们团队有底气去做 MVVM 重构、去坚持严格分层、去写一份与代码同步更新的架构文档。

这个过程让团队对"软件工程"这四个字有了更深的理解——**它不是关于你写了多少行代码，而是关于你写的代码在多大程度上可被理解、可被测试、可被扩展**。
